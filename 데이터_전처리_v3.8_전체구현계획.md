# 배터리 이미지 데이터 전처리 v3.8 전체 구현 계획

> 작성일: 2026-07-20
> 수정일: 2026-07-20
> 상태: 대화 검토 결정 반영 수정본
> 대상: AIHub 배터리 불량 이미지 CT·RGB 원본 데이터
> 출력: YOLOv11 detection·segmentation 학습 데이터, 원본 JSON 보존본, manifest, 품질 보고서
> 기본 seed: `42`
> 기준 Python: `3.12`
> 필수 라이브러리: `Pillow==12.2.0`, `shapely==2.1.2`

> **원본 불변조건:** 전처리 과정은 원본 이미지와 원본 JSON을 읽기 전용으로 취급한다. 원본을 수정·삭제·이동·이름 변경하지 않으며, 실행 전후 경로·크기·SHA-256 불변성을 감사한다. 모든 제외는 split 모집단과 파생 학습 출력에만 적용한다.

이 문서는 v3.4 또는 v3.5 계획서를 읽지 않아도 전체 전처리 코드를 구현할 수 있도록 이전 버전에서 확정된 규칙과 v3.8 변경사항을 모두 합친 독립 문서다. 이 문서와 이전 문서가 다르면 이 문서를 우선한다.

> **v3.8 최우선 변경:** CT의 복구·ROI clipping·multipart 분리 후 유효 porosity 조각 중 `porosity_bbox_max_ratio >= 0.25`인 조각이 하나라도 있는 이미지는 geometry 계산과 lineage 기록 후, Battery ID 통계·Test/development 선택·5-fold 층화보다 먼저 정책 제외한다. bbox 비율은 소수점 8자리로 직렬화한 YOLO 정규화 좌표에서 계산한다. 이후 모든 split 비율과 품질 지표는 제외 후 모집단으로 다시 계산한다. 이 규칙은 v3.6 §23의 “대용량 라벨 유지” 결정을 명시적으로 대체한다.

계획서와 실제 전달 코드에서 다음 이름을 고정한다. 이전 내부 패키지명은 사용하지 않는다.

## 1. 목표와 범위

### 1.1 목표

원본 CT·RGB 데이터 전체를 스캔해 다음을 수행한다.

1. 이미지와 JSON을 정확히 1:1로 매칭한다.
2. 파일명, 배터리 ID, 영상 크기, CT ROI, JSON 구조를 검증한다.
3. 손상 이미지와 잘못된 라벨을 분리한다.
4. 동일 ID 내부 중복과 서로 다른 ID 사이의 pixel 중복을 처리한다.
5. 원본 JSON은 수정하지 않고 YOLO detection·segmentation 라벨을 파생 생성한다.
6. 배터리 ID 기준으로 train·validation·test를 분할해 누수를 차단한다.
7. CT는 Test 7 ID와 development 40 ID로 구성하고 5-fold CV를 만든다.
8. CT는 개별 porosity bbox 최대 면적비가 25% 이상인 이미지를 선제 제외하고, development/Test 모두 남은 유효 이미지 전량을 사용한다.
9. CT fold에 결함 비율·annotation 밀도·축·porosity 면적합 분포를 균형 배치한다.
10. RGB는 대표 160 ID를 선정해 128/16/16 ID로 분할한다.
11. 모든 선택·제외·변환 결과를 manifest와 CSV로 추적할 수 있게 한다.

### 1.2 “전체 데이터 출력”의 의미

전체 원본을 검사하지만 원본 파일을 무조건 모두 복사하지는 않는다. 계획서의 유효성·중복·대표성 기준을 통과한 **최종 학습 대상 전체**를 출력한다.

- CT: 47개 유효 배터리 ID. Test 7 ID와 development 40 ID의 모든 유효 이미지
- RGB: 대표 160 ID. Train/Validation은 ID별 최대 300장, Test는 전량
- 제외 대상: 손상, 매칭 실패, 정의되지 않은 클래스, 잘못된 ROI, cross-ID pixel 중복, 학습 불가능 라벨 등
- 제외 사유는 삭제하지 않고 manifest와 이슈 CSV에 기록한다.

### 1.3 비목표

- 원본 JSON 수정
- CT ROI 외 영역을 포함한 학습
- RGB ROI crop
- 배터리 ID를 여러 split에 나누는 이미지 단위 랜덤 분할
- 원본 클래스명 임의 추가
- 증강 데이터 생성
- 모델 학습 자체

## 3. 입력 데이터와 파일 탐색

### 3.1 입력 루트

`--raw-root`는 Training과 Validation의 원천·라벨 데이터를 모두 포함하는 상위 폴더다.

예시:

```text
E:\103.배터리 불량 이미지 데이터\3.개방데이터\1.데이터
```

특정 `TS_*`, `TL_*` 폴더명을 코드에 하드코딩하지 않는다. `raw-root` 아래를 재귀 탐색한다.

### 3.2 지원 확장자

- 이미지: `.jpg`, `.jpeg`, `.png`, `.bmp`, `.tif`, `.tiff`
- 라벨: `.json`
- 확장자 비교는 case-insensitive로 처리한다.

### 3.3 이미지·JSON 매칭

Training/Validation 같은 원본 구역과 파일 stem을 이용해 매칭한다.

```text
pair_key = (원본 구역, 파일 stem casefold)
```

원천·라벨 폴더의 상위 경로 구조가 서로 다를 수 있으므로 전체 상대경로가 같다고 가정하지 않는다.

다음 조건을 모두 만족해야 1:1 매칭이다.

- 동일 pair key의 이미지 1개
- 동일 pair key의 JSON 1개
- 후보 이미지 또는 JSON이 0개이거나 2개 이상이면 매칭 실패
- 실패 항목은 `matching_issues.csv`에 기록

### 3.4 파일명 문법

CT:

```text
CT_cell_<form>_<battery_id>_<axis>_<frame>
axis ∈ {x, y, z}
```

RGB:

```text
RGB_cell_<form>_<battery_id>_<frame>
```

배터리 ID는 파일명에서 파싱한다. JSON의 `image_info.id`를 배터리 ID로 사용하지 않는다. 숫자 ID의 선행 0은 제거한다.

파일명이 문법과 맞지 않으면 해당 샘플을 제외하고 `matching_issues.csv`에 기록한다.

### 3.5 sample ID

출력 파일명 충돌을 방지하기 위해 원본 stem과 정규화 상대경로를 기반으로 결정론적 `sample_id`를 만든다.

```text
sample_id = <modality>__<sanitized_original_stem>__<stable_hash_prefix>
```

동일 입력에서 항상 같은 sample ID가 생성되어야 한다.

## 4. JSON 파싱과 클래스 표준화

### 4.1 JSON 주요 필드

구현은 최소한 다음 필드를 읽는다.

```text
data_info.data_type
data_info.application
data_info.roi
image_info.width
image_info.height
image_info.is_normal
defects[]
defects[].name
defects[].points
```

실제 JSON의 중첩 위치가 일부 다를 수 있으므로 기존 스키마에서 확정된 후보 위치를 탐색하되, 모호한 값을 임의 추정하지 않는다.

### 4.2 modality

- `ct` → `CT`
- `rgb`, `ext`, `exterior` → `EXT`
- 그 외 → `UNKNOWN`으로 기록 후 제외

파일명 modality와 JSON modality가 다르면 어느 한쪽을 임의로 신뢰하지 않는다. `modality_mismatch`로 `json_anomalies.csv`와 manifest에 기록하고 detection·segmentation 모두에서 제외한다.

### 4.3 클래스 별칭

비교 전 `_`와 `·`를 공백으로 바꾸고 연속 공백을 축약한 뒤 casefold한다.

| modality | 표준 클래스 | 허용 별칭 |
|---|---|---|
| CT | `porosity` | `porosity` |
| EXT | `Damaged` | `damaged`, `damage` |
| EXT | `Pollution` | `pollution`, `contamination` |

`battery_outline`은 학습 결함 클래스가 아니므로 무시한다.

정의되지 않은 클래스가 있는 이미지는 학습 라벨의 의미가 불명확하므로 해당 이미지를 제외하고 `class_name_variants.csv`에 기록한다.

### 4.4 정상 이미지

- `is_normal == true`이고 학습 대상 결함이 없으면 정상 이미지로 포함한다.
- `is_normal == true`인데 결함이 존재하면 이상으로 기록하지만 유효 결함 라벨 검사는 계속 수행한다.
- 대상 결함도 없고 명시적 정상도 아니면 제외한다.
- 정상 이미지의 YOLO TXT는 빈 파일이다.

### 4.5 defects 배열 내부 중복

동일 JSON 안에서 `(표준 클래스명, 정규화 points)`가 완전히 같은 defect가 반복되면 첫 항목만 유지한다.

- 원본 JSON은 수정하지 않는다.
- 중복 제거는 파생 YOLO 라벨에만 적용한다.
- 제거 개수는 `json_anomalies.csv`와 manifest에 반영한다.

## 5. 이미지·ROI·좌표 검증

### 5.1 이미지 디코딩

Pillow로 이미지를 실제 디코딩한다.

- 열 수 없는 이미지
- 폭 또는 높이가 0 이하
- 디코딩 예외

위 항목은 `corrupt_images.csv`에 기록하고 제외한다.

### 5.2 좌표 프레임

CT:

- 프레임 폭·높이 = `data_info.roi`
- 원본 이미지에서 좌상단 기준 ROI 크기로 crop
- YOLO 좌표도 ROI 기준으로 정규화

RGB:

- crop하지 않음
- 프레임 폭·높이 = 전체 이미지 크기
- YOLO 좌표는 전체 이미지 기준으로 정규화

### 5.3 CT ROI 검증

다음을 모두 만족해야 한다.

- 길이 2의 배열
- width·height가 유한한 양수
- 원본 이미지 범위를 넘지 않음

실패 시 `invalid_ct_roi`로 제외한다.

### 5.4 points 형식

다음 두 형식을 허용한다.

```text
[x1, y1, x2, y2, ...]
[[x1, y1], [x2, y2], ...]
```

조건:

- 최소 서로 다른 점 3개
- 모든 좌표가 숫자이며 finite
- flat 배열은 길이가 짝수
- 마지막 점이 첫 점과 같으면 마지막 중복점 제거

### 5.5 폴리곤 복구와 clipping

Shapely를 사용한다.

1. `Polygon(points)` 생성
2. self-intersection이면 `buffer(0)`으로 복구 시도
3. 프레임 밖 부분은 frame box와 intersection
4. MultiPolygon이면 유효 조각별로 분리
5. 양의 면적을 가진 조각만 segmentation에 사용
6. 조각마다 bbox를 만들어 detection에 사용

원본 점이 유효하고 프레임 안에 완전히 있으면 원래 점 순서를 유지한다. 복구·clipping된 점은 결정론적으로 시작점과 방향을 정규화한다.

### 5.6 detection과 segmentation 분리

얇은 선형 라벨처럼 면적 0이지만 경계가 존재하는 경우:

- segmentation: 제외
- detection: bbox 폭 또는 높이가 0이면 중심 기준 최소 1px extent 적용
- `min_extent_applied=true` 기록

따라서 detection과 segmentation의 이미지 수는 다를 수 있다.

이미지 내 대상 defect 중 하나라도 detection 변환에 실패하면 그 이미지의 detection 라벨 전체를 제외한다. segmentation도 동일 원칙을 별도로 적용한다.

## 6. 중복과 누수 처리

### 6.1 pixel hash

파일 byte가 아니라 디코딩된 RGB pixel을 기준으로 해시한다.

```text
pixel_hash = SHA-256(width || height || RGB pixel bytes)
```

JPEG 메타데이터나 저장 방식만 다른 동일 이미지를 잡기 위한 규칙이다.

### 6.2 같은 ID 내부 중복

동일 modality·battery ID 안에서 pixel hash가 같은 경우 결정론적으로 대표 1개를 유지한다.

대표 선정:

```text
(stem casefold, image_path casefold) 오름차순 첫 항목
```

나머지는 duplicate로 제외하고 `pixel_duplicates.csv`에 기록한다.

### 6.3 서로 다른 ID 사이 중복

동일 pixel hash가 둘 이상의 battery ID에 나타나면 라벨 충돌과 누수 위험이 있으므로 해당 hash 그룹 전체를 제외한다.

### 6.4 최종 누수 감사

모든 split/fold 쌍에 대해 다음 교집합을 계산한다.

- battery ID 교집합
- pixel hash 교집합

한 건이라도 있으면 execute를 중단한다.

## 7. 공통 지표

모든 모듈은 하나의 공통 함수로 아래 지표를 계산해야 한다. 선택·swap·게이트·보고서가 서로 다른 정의를 사용하면 안 된다.

```text
images = 이미지 수
defect_images = detection 결함 인스턴스가 1개 이상인 이미지 수
annotations = detection 인스턴스 총수
defect_image_ratio = defect_images / images
annotations_per_image = annotations / images
normal_ratio = 정상 이미지 수 / images
```

빈 입력의 비율은 모두 `0.0`으로 정의한다.

`defect_image_ratio`는 1차 분할 지표다. `annotations_per_image`는 v2에서 발생했던 Validation 0.41 대 Training 0.19의 2.2배 밀도 차이를 막는 2차 지표다.

## 8. CT porosity 면적합 지표와 25% bbox 제외 지표

### 8.1 계산식

폴리곤 면적은 shoelace 공식 또는 Shapely polygon area를 사용한다. 실제 학습에 포함되는 clipping·복구 후 유효 폴리곤 기준으로 계산한다.

```text
porosity_area_sum_ratio
= 이미지 내 유효 porosity polygon area 합계 / (roi_width × roi_height)
```

한 이미지에 porosity가 여러 개면 모두 더한다. 이미지 장수는 한 장으로 센다.

### 8.2 porosity가 없는 이미지

정상 CT 또는 porosity가 없는 유효 CT도 반드시 포함한다.

```text
porosity_polygon_count = 0
porosity_area_sum_ratio = 0
porosity_area_bin = zero
```

양성 CT만 입력하면 정식 층화가 아니다.

### 8.3 면적 구간

경곗값은 하한 포함·상한 미포함이다.

| 구간명 | 범위 |
|---|---:|
| `zero` | 0 |
| `lt_0.1pct` | 0 초과 ~ 0.1% 미만 |
| `0.1_1pct` | 0.1% 이상 ~ 1% 미만 |
| `1_5pct` | 1% 이상 ~ 5% 미만 |
| `5_10pct` | 5% 이상 ~ 10% 미만 |
| `10_25pct` | 10% 이상 ~ 25% 미만 |
| `25_40pct` | 25% 이상 ~ 40% 미만 |
| `40_50pct` | 40% 이상 ~ 50% 미만 |
| `ge_50pct` | 50% 이상 |

```text
ct_stratum = axis + "|" + porosity_area_bin
```

폴리곤이 서로 겹치면 단순 합에서 중복 면적이 발생할 수 있다. 현재 EDA 정의와 맞추기 위해 단순 합을 사용하고, 비율이 비정상적으로 큰 이미지는 검수 CSV에 남긴다.

면적합과 별도로, 노트북 `battery_ct_tiled_v36 (1).ipynb`의 25% 판정을 재현하기 위한 bbox 지표를 계산한다.

```text
porosity_bbox_ratio(component)
= (bbox_width / roi_width) × (bbox_height / roi_height)

porosity_bbox_max_ratio
= 모든 유효 porosity component의 porosity_bbox_ratio 최댓값
```

- 원본 points를 바로 사용하지 않고 polygon 복구·ROI clipping·MultiPolygon 분리 후의 유효 조각을 사용한다.
- 각 조각의 좌표는 YOLO segmentation 출력과 동일하게 ROI 기준 `[0,1]`로 정규화하고 소수점 8자리로 반올림한 뒤 bbox를 계산한다.
- porosity가 없는 유효 CT는 `porosity_bbox_max_ratio=0.0`이다.
- `porosity_area_sum_ratio`와 `porosity_area_bin`은 EDA·fold 층화용이며 25% 정책 제외 판정에는 사용하지 않는다.

### 8.4 v3.8 선제 제외 정책

다음 조건을 만족하는 CT 이미지는 split 모집단에서 제외한다.

```text
modality == CT
and porosity_bbox_max_ratio >= 0.25
```

- `0.25` 경계는 제외에 포함한다.
- `0.25`는 소수점 8자리로 반올림한 정규화 좌표에서 계산한 개별 유효 조각 bbox 면적비 기준이다.
- 유효 조각이 여러 개면 합산하지 않고 최댓값을 사용한다.
- 원본 이미지와 JSON을 물리적으로 삭제하지 않는다.
- `manifest.csv`에 `pre_split_eligible=false`와 제외 사유를 남긴다.
- `ct_bbox_exclusions.csv`에 제외 이미지를 전수 기록한다.
- 40% 이상 행은 `ct_large_area_review.csv`에도 계속 남긴다.
- 제외가 끝난 뒤 ID별 이미지 수, 결함률, annotation 밀도, 축 및 잔여 면적구간 통계를 새로 계산한다.
- 한 ID의 CT 이미지가 모두 제외되어 잔여 CT ID가 47개 미만이면 구조 게이트로 중단한다.

## 9. CT Test와 5-fold 배정

### 9.1 구조

- 유효 CT ID: 정확히 47
- Test: 7 ID
- Development: 40 ID
- Fold: 5개
- 각 fold validation: 8 ID
- 각 fold training: 나머지 32 ID
- Test는 모든 fold에서 동일하게 고정

### 9.2 CT 이미지 선택

Test/development를 정하기 전에 §8.4의 개별 porosity bbox 최대 면적비 25% 이상 이미지를 전체 CT 모집단에서 먼저 제외한다. Test와 development는 제외 후 남은 유효 이미지를 전량 사용한다.

- x/y/z 각각 100장 제한 없음
- ID별 300장 제한 없음
- 복제 없음
- 부족한 축을 다른 축으로 충당하지 않음
- 자연 축 분포를 보존
- 제외 전 통계나 승인 CSV를 재사용하지 않음

### 9.3 고정 Test ID

승인된 `test_battery_ids.csv`가 있더라도 v3.8 전용 work-dir, raw fingerprint, seed, 제외 후 ID 통계 fingerprint와 승인 정책(`metric=porosity_bbox_max_ratio`, `precision=8`, `operator=>=`, `threshold=0.25`)이 모두 현재 실행과 일치할 때만 정확히 7 ID인지 확인하고 잠근다. 하나라도 다르면 승인본을 거부하고 dry-run부터 다시 수행한다. fold 균형을 위해 유효한 승인 Test ID를 바꾸지 않는다.

### 9.4 신규 Test 선정

승인본이 없을 때만 선정한다. bbox 최대 면적비 25% 이상 이미지가 제거된 ID 통계로 후보 Test와 나머지 development의 차이를 다음 우선순위로 최소화한다.

```text
score =
    3 x axis_max_gap
  + 1 x axis_sum_gap
  + 6 x defect_image_ratio_relative_gap
  + 6 x annotations_per_image_relative_gap
  + 1 x remaining_porosity_area_bin_max_gap
```

동점일 때만 deterministic_tie_key로 순서를 정한다.

가중 합을 쓰는 이유는 사전식 튜플이 실패했기 때문이다. v3.8 초기 구현은 위 항목들을 순서대로 비교했는데, 앞선 항목이 실수값이라 동점이 거의 발생하지 않아 밀도 항까지 비교가 내려오지 않았다. 그 결과 밀도 상위 ID가 Test에 몰려 Test/development 밀도비가 2.78배가 됐다. 가중 합으로 바꾼 뒤 밀도비 0.947배, defect 비율 1.001배가 됐다. defect 격차도 절대차 대신 상대차로 계산해 밀도 격차와 스케일을 맞춘다.

1. greedy로 7 ID를 채운다.
2. Test 1 ID와 development 1 ID의 모든 pairwise swap을 평가한다.
3. 목적함수가 엄격히 감소하는 최선의 swap을 반복한다.
4. 개선이 없으면 종료한다.

### 9.5 development 5-fold 층화

ID를 쪼개지 않고 40 ID를 8개씩 배정한다. 각 ID에서 선제 제외 후 남은 데이터로 다음 통계를 미리 계산한다.

- 이미지 수
- defect image 수와 비율
- annotation 수와 이미지당 밀도
- x/y/z 이미지 수
- 면적 구간별 이미지 수
- 축×면적 구간별 이미지 수

목적함수:

```text
score =
  12 × 전체 이미지 장수 정규화 오차²
  + 8 × defect_image_ratio 정규화 오차²
  + 4 × annotations_per_image 정규화 오차²
  + 3 × 축별 장수 정규화 오차²
  + 2 × 축×면적구간 장수 정규화 오차²
  + 1 × 면적구간 장수 정규화 오차²
```

각 fold의 목표는 development 전체의 1/5이다. seed 42 기반 결정론적 다중 재시작 greedy 탐색을 500회 수행하고 합계 score가 최소인 배정을 채택한다.

### 9.6 밀도 swap

초기 fold 배정 후 `annotations_per_image` 편차를 줄인다.

- CT에는 stratum 동일 제약을 적용하지 않는다. 밀도가 두꺼운 ID가 인구 희박한 stratum에 갇히면 교환 상대가 없어 편차를 전혀 줄이지 못한다. 실측으로 최악 fold 편차가 +26.93%(게이트 실패)에서 +13.39%로 바뀌었고, defect 균형은 아래 ratio_spread 가드가 지켜 이탈이 0.0406에서 0.0426으로만 움직였다.
- RGB(§10.5)에는 stratum 제약을 유지한다. stratum = `round_ties_to_even(defect_image_ratio(id) × 10)`
- 교환 후 defect ratio 최대 편차가 기존보다 0.01 초과 악화되면 기각
- annotation 밀도 최대 편차가 감소하는 교환만 수락
- 15% 이내가 되면 종료
- 최대 12라운드
- 최종 ±20%를 넘으면 execute 중단

### 9.7 CT 면적 분포 경고

- fold 이미지 비율이 목표 20%에서 5%p 이상 벗어나면 경고
- 특정 면적 구간 이미지의 40% 이상이 한 fold에 몰리면 경고
- 선택 모집단에 `porosity_bbox_max_ratio >= 0.25` 이미지가 한 장이라도 남으면 하드 게이트 실패
- 제외된 bbox 25% 이상 이미지는 `ct_bbox_exclusions.csv`에 전수 기록
- 40% 이상 제외 이미지는 `ct_large_area_review.csv`에도 기록

경고는 ID 누수를 허용하는 근거가 아니다.

## 10. RGB ID 선정과 split

### 10.1 대표 160 ID 선정

모든 유효 RGB ID에서 결정론적으로 최대 300장 가상 표본을 만든다. 다음 strata로 160 ID를 비례 선정한다.

```text
(application, has_damaged, defect_image_ratio 5분위)
```

`defect_image_ratio`는 각 ID의 최대 300장 가상 표본에서 계산한다. ID를 `(defect_image_ratio, stable_key)`로 정렬한 뒤 ID 수가 최대한 같도록 순위 기반 5개 구간에 배정한다. 동일 비율이 경계에 몰려도 stable key로 결정론적으로 나누어 구간 크기 균형을 우선한다. strata별 160 ID quota는 largest-remainder 방식으로 계산하고, 잔여 quota 동률은 stable hash 순으로 배정한다.

application 표준화:

- `home`, `consumer`, `appliance`, `가전` → `가전`
- `industrial`, `industry`, `ev`, `산업` → `산업`
- 빈 값은 빈 값 유지

ID 내부 application이 섞이면 다수결, 동수면 사전순 최소값을 사용하고 경고한다.

### 10.2 대표성 게이트

선정 160 ID가 raw 유효 RGB 모집단 대비 다음을 만족해야 한다.

- application 비율 차이: 각 범주 ±2%p
- defect_image_ratio 차이: ±3%p
- has_damaged ID 비율 차이: ±2%p
- 선정 이미지 정상 비율: 최소 30%

실패 시 중단한다.

### 10.3 RGB split

- Train: 128 ID
- Validation: 16 ID
- Test: 16 ID

동일 strata 기준으로 Test를 먼저 선정하고, 나머지에서 Validation을 선정한다. 승인된 Test 16 ID가 있으면 잠근다.

Validation과 Test 각각에 Damaged 보유 ID가 최소 1개 있어야 한다.

### 10.4 RGB 이미지 장수

- Train/Validation: ID별 최대 300장
- Test: 유효 이미지 전량
- 300장 표본은 정상·결함 비율을 보존하는 결정론적 층화 표본

### 10.5 RGB 밀도 보정

Test는 잠그고 Train과 Validation 사이에서만 동일 strata ID swap을 수행한다.

- target = Train+Validation 전체 `annotations_per_image`
- swap 목표 = 15%
- 최종 허용 = ±20%
- Test 밀도는 보고만 하고 게이트에서 제외

  --raw-root "E:\103.배터리 불량 이미지 데이터\3.개방데이터\1.데이터" `

7. 실패 시 최종 output을 만들지 않고 실패 보고서 보존

## 17. 품질 게이트

### 17.1 즉시 중단하는 구조 게이트

- CT 유효 ID가 47개가 아님
- CT Test 7·development 40 구조 불일치
- CT fold가 8/8/8/8/8 ID가 아님
- RGB split이 128/16/16 ID가 아님
- CT axis가 x/y/z 이외
- 이미지·JSON 매칭 모호성으로 필수 구조를 만들 수 없음
- split/fold 간 battery ID 교집합 존재
- split/fold 간 pixel hash 교집합 존재
- 승인본과 재계산 ID 배정 불일치
- raw fingerprint 변경
- 원본 JSON SHA-256 불일치
- 출력 역검증 실패
- 배정되지 않은 선택 CT ID 또는 이미지 존재
- CT 면적합이 음수 또는 non-finite
- manifest와 CT 분포 CSV 이미지 합계 불일치

### 17.2 승인 필수 품질 경고

- CT 5개 fold의 `annotations_per_image`가 development 전체 target 대비 ±20% 초과
- RGB Train/Validation의 `annotations_per_image`가 Train+Validation target 대비 ±20% 초과
- det 포함 이미지 대비 seg 제외율 7% 초과
- det와 seg `defect_image_ratio` 차이 3%p 초과
- Validation/Test 필수 클래스 부재
  - CT 각 validation fold와 Test: porosity
  - RGB Validation/Test: Damaged와 Pollution

고정 Test의 annotation 밀도는 보고만 하고 ±20% 게이트에서 제외한다.

이 절의 경고는 일단 승인과 execute를 중단하지만, 검토자가 `quality_exceptions.csv`에 `status=approved_exception`, `reviewer`, `reviewed_at`, `reason`을 기록하면 진행할 수 있다. execute 시 warning ID와 관측값이 dry-run 결과와 달라지면 승인은 무효다. 이 예외 절차는 §17.1 구조·무결성 게이트에는 적용되지 않는다.

### 17.3 검토 경고

- Test ID 유효 이미지 100장 미만
- Test ID 원본 대비 제외율 20% 이상
- CT fold 이미지 비율 20% 목표에서 5%p 이상 차이
- CT 특정 면적 구간의 40% 이상이 한 fold에 집중
- CT 면적합 40% 이상 이미지
- ID 내부 application 혼재
- 폴리곤 복구·multipart·min-extent 다수

경고는 검토와 승인 근거를 기록하지만 실행을 자동 중단하지 않는다. 이 절의 경고를 실행 시 임의로 하드 게이트로 승격하지 않는다. 등급 변경은 계획서 버전 개정과 회귀 테스트를 거쳐야 한다.

## 23. v3.6 대용량 라벨 처리 결정 기록 — v3.8에서 대체됨

### 23.1 배경

AI팀의 대용량 라벨 문의를 계기로 v3.6에서 CT porosity 면적합 지표와 검수 절차를 도입했다. v3.6은 자동 제외 대신 식별과 확인을 선택했다. 이 절은 이력 보존용이며 v3.8 실행 정책은 §8.4가 우선한다.

### 23.2 검토한 두 선택지

- (a) 자동 제외 또는 캡: 면적합이 큰 라벨이나 이미지를 학습에서 제외하거나 detection bbox 크기에 상한을 둔다.
- (b) 유지 + 분산 + 기록: 대용량 라벨을 그대로 학습에 사용하되 면적합 지표로 fold 간 균형을 맞추고 검수 목록에 전수 기록해 확인할 수 있게 한다.

### 23.3 v3.6 당시 결정: (b) 유지 + 분산 + 기록

v3.6 당시에는 대용량 라벨을 제외하지 않고 학습에 사용했다. v3.8에서는 사용자 정책 변경에 따라 개별 porosity bbox 최대 면적비 25% 이상 이미지를 split 전에 제외한다.

- v3.6에서는 면적합 지표를 fold 균형·검수 목적으로 사용했다.
- v3.8에서는 `porosity_bbox_max_ratio >= 0.25`를 선제 제외 기준으로 사용한다. 면적합 구간은 EDA·층화에만 사용한다.
- 면적합 40% 이상 이미지는 `ct_large_area_review.csv`에 전수 기록해 AI팀이 "큰 박스" 사례를 직접 확인할 수 있게 한다.
- v3.8에서도 제외 후 잔여 면적구간 쏠림은 검토 경고로 유지하되, bbox 최대 면적비 25% 이상 이미지가 선택 모집단에 남는 것은 하드 게이트다.

### 23.4 참고: 면적합과 bbox 크기의 차이

면적합은 폴리곤 면적을 기준으로 하며 detection bbox 크기와 정확히 같지는 않다. 가늘고 긴 대각 폴리곤은 면적이 작아도 bbox가 클 수 있고, 겹치는 폴리곤은 면적합이 이중으로 계산되어 실제보다 크게 잡힐 수 있다. 박스 크기 자체를 정밀하게 검수하려면 개별 detection bbox extent를 별도로 산출해 보완한다.

## 24. 트러블슈팅: 면적 쏠림 경고가 execute를 영구 차단한 문제 (2026-07-17 18:11 수정)

> **v3.6 이력 전용:** 이 절의 증상·수치·변경 파일은 v3.6 당시 장애 기록이며 v3.8 실행 명세가 아니다. v3.8의 현재 게이트 등급과 승인 절차는 §17 및 §13.2를 따른다. 이 절을 근거로 bbox 25% 제외 정책, 품질 예외 승인, 보고서 라우팅을 변경하지 않는다.

### 24.1 증상

전체 raw 데이터로 dry-run을 실행하면 모든 구조 게이트와 하드 품질 게이트를 통과하지만, `approve-selection`이 `cannot approve: 3 dry-run quality warning(s)` 오류로 실패했다. `execute` 역시 같은 경고에서 중단되도록 되어 있었다. 차단 원인은 다음 세 경고였다.

```
CT_area_bin[10_25pct]: >=40% concentrated in one fold
CT_area_bin[40_50pct]: >=40% concentrated in one fold
CT_area_bin[5_10pct]: >=40% concentrated in one fold
```

### 24.2 근본 원인

- 이 경고들은 계획서 §17.3의 검토 경고이며, §17.3은 "구조 게이트와 동일한 자동 중단은 아니다"라고 규정한다.
- 그러나 기존 코드의 `reports.quality_warnings()`는 §17.2 하드 게이트(밀도 ±20%, 필수 클래스, det/seg)와 §17.3 검토 경고(면적 구간 쏠림, fold 이미지 비율, Test 100장 미만, Test 제외율)를 하나의 리스트로 합쳐 `dryrun_warnings.csv`에 기록했다.
- `pipeline.approve_selection`과 `pipeline.execute`는 `dryrun_warnings.csv`가 비어 있지 않으면 무조건 중단하며, 이 채널에는 acknowledge(검토 확인) 경로가 없었다. acknowledge 경로는 `review_warnings.csv`에만 존재한다.
- 게다가 면적합 40% 이상 이미지 890장이 전부 battery ID 134에 몰려 있고, §22.1이 같은 ID를 여러 fold로 나누는 것을 금지하므로, ID 134가 배정된 fold가 `40_50pct` 구간을 100% 독점한다. 따라서 면적 쏠림 경고는 어떤 배정에서도 반드시 발생하며 swap으로 제거할 수 없다. 결과적으로 전달 코드로는 execute가 절대 통과하지 못하는 상태였다.

### 24.3 수정 (계획 §17.2/§17.3 분류를 코드로 반영)

경고를 삭제하거나 우회하지 않는다(§22.8). 대신 계획서가 이미 정의한 하드·소프트 분류를 코드로 정확히 구현한다.

- `selection.SelectionResult`에 `review_warnings` 필드를 추가하고, 면적 균형 검사(`_check_ct_area_balance`) 결과를 `warnings`(하드) 대신 여기에 담는다. 밀도 게이트(`_check_density`) 결과는 하드로 유지한다.
- `reports.quality_warnings()`가 `(gate, review)` 두 리스트를 반환한다. gate는 §17.2 하드 게이트(밀도, 필수 클래스, det/seg), review는 §17.3 검토 경고(면적 쏠림, fold 이미지 비율, Test 100장 미만, Test 제외율)다.
- `reports.write_reports()`는 gate를 `dryrun_warnings.csv`에, review를 scan 단계 review 경고와 합쳐 `review_warnings.csv`에 `status=pending`으로 기록한다.
- `pipeline`은 변경하지 않는다. `approve_selection`·`execute`는 `dryrun_warnings.csv`(이제 하드 게이트만)로 하드 블록하고, `review_warnings.csv`의 모든 행이 `acknowledged`여야 진행한다는 기존 로직을 그대로 사용한다.

### 24.4 수정 후 승인 절차

1. dry-run을 다시 실행하면 `dryrun_warnings.csv`는 비고, 면적 쏠림 경고 3건은 `review_warnings.csv`에 `pending`으로 남는다.
2. 검토자가 각 review 경고를 확인하고 `review_warnings.csv`의 `status`를 `acknowledged`로, `reviewer`·`reviewed_at`을 채운다. 이 항목은 v3.6 이력 설명이며 v3.8의 승인 필수 품질 경고는 `quality_exceptions.csv`를 사용한다.
3. `approve-selection` → `execute` 순서로 진행한다. 모든 경고는 `review_warnings.csv`에 감사 기록으로 남으므로 숨기지 않으며, 누수 게이트는 완화하지 않는다.

## 26. 트러블슈팅: CT fold 밀도 게이트 실패와 ID 게이트 도입 (2026-07-20 수정)

### 26.1 증상

v3.8의 전신인 v3.7로 dry-run을 완주하자 `dryrun_warnings.csv`에 하드 게이트 경고 2건이 남아 execute가 막혔다. v3.6에서는 같은 경고가 0건이었다.

```text
CT_fold:0: annotations_per_image deviation=0.52461203 target=0.20709406 achieved=0.31573810
CT_fold:1: annotations_per_image deviation=0.25529740 target=0.20709406 achieved=0.15422349
```

fold별 이미지 장수는 목표 20%에서 최대 3.3%p 차이로 균형이 나쁘지 않았다. 깨진 것은 장수가 아니라 annotation 밀도였다.

### 26.2 원인

원인은 fold 배정 알고리즘이 아니라 §8.4 이미지 단위 제외 정책이었다.

CT 배터리 ID별 annotation 밀도는 꼬리가 두꺼운 분포다. 제외 전 기준으로 ID 109가 전체 CT annotation의 21.3%를 이미지의 2.4%에 담고 있었다. 여기에 bbox 최대 면적비 25% 이상 제외를 적용하면 8,345장이 사라지는데, 그 제거가 ID별로 극도로 비대칭이다.

| fold | 제외 장수 | 밀도 편차 |
|---|---|---|
| 0 | 3 | +52.5% |
| 1 | 2,743 | -25.5% |
| 2 | 1,510 | -12.6% |
| 3 | 1,440 | -12.5% |
| 4 | 1,833 | -14.2% |

제외 대상은 대형 공동이 찍힌 이미지, 즉 annotation을 가진 이미지다. fold 1~4는 annotation이 대량으로 깎였지만 ID 109가 속한 fold 0은 3장만 잃었다. 그 결과 109의 annotation 점유율이 18.7%에서 30.7%로 올라가고, 전체 목표 밀도는 내려가면서 fold 0만 상대적으로 튀어올랐다.

### 26.3 fold 재배정으로는 해결되지 않는다

40개 ID를 5개 fold에 배정하는 모든 조합을 지역 탐색으로 훑어 최악 fold 편차를 최소화한 결과는 +35.12%였다. 해석적 하한과 정확히 일치했고, ±20% 게이트에 도달하는 배정은 존재하지 않는다.

이유는 단순하다. ID 109는 1,617장에 annotation 3,626개를 갖는다. development 전체 annotation은 11,823개이므로 fold 하나의 목표치는 2,365개인데, ID 109 하나가 그 예산을 1.5배 초과한다. 8-ID 정원 안에서 분모(이미지)를 최대로 키워도 12,958장이 한계라 밀도 하한이 0.2798, 즉 목표 대비 +35%다.

§9.6 밀도 swap의 stratum 제약을 푸는 방법도 시험했다. 최악 편차는 +52.46%에서 +43.17%로 줄었지만 게이트는 여전히 통과하지 못했고, fold 간 defect_image_ratio 최대 이탈이 0.0148에서 0.0386으로 나빠졌다. 밀도 편차를 줄이려면 109를 annotation이 없는 ID들과 같은 fold에 몰아야 하는데, 그 ID들은 defect 비율이 0이라 1차 지표가 무너진다. 이 데이터에서 밀도 균형과 defect 균형은 정면으로 충돌한다.

### 26.4 임계값 조정으로도 해결되지 않는다

AI팀에서 제외 임계값을 25%에서 5%로 낮추는 방안이 논의되어 함께 검증했다.

| 임계값 | CT 잔존 | annotation 잔존 | fold 하한 |
|---|---|---|---|
| 제외 없음 | 100% | 100% | +0.03% |
| 0.25 | 89.0% | 67.1% | +35.12% |
| 0.10 | 89.0% | 67.1% | +35.15% |
| 0.05 | 88.2% | 64.6% | +37.00% |
| 0.03 | 84.5% | 50.8% | +19.49% |

임계값을 낮춰도 나아지지 않고 오히려 악화된다. ID 109의 기공은 잘고 많아서 bbox 최대 면적비가 0.05를 넘는 이미지가 3장뿐이다. 임계값을 조여도 109는 그대로 남고 다른 ID만 더 깎이므로 109의 점유율이 올라간다.

bbox 최대 면적비 분포가 이봉형이라는 점도 확인했다. 5~15% 구간에 625장, 15~25% 구간에 1장뿐이고 25% 이상이 8,345장이다. 따라서 임계값을 0.05와 0.25 사이 어디에 두든 결과는 626장 차이에 그친다.

### 26.5 annotation 잔존율은 크기별로 봐야 한다

"annotation 67.1% 잔존"이라는 단일 수치는 오해를 부른다. 기공 크기별로 나누면 25% 정책이 무엇을 버리는지 분명해진다.

| 정책 | 미세 <0.5% | 소 0.5-2% | 중 2-5% | 대 5-15% | 초대 >=15% |
|---|---|---|---|---|---|
| 이미지 >=0.25 | 100% | 100% | 100% | 100% | 0% |
| 이미지 >=0.05 | 100% | 99.4% | 64.7% | 0% | 0% |

25% 정책은 미세부터 대형까지 전부 보존하고 초대형만 제거한다. 학습 목표가 미세 결함이라면 손실이 없다. 반대로 임계값을 0.05로 낮추면 중형 기공 annotation이 35% 사라진다. 미세 결함 학습에는 25%가 유리하다.

### 26.6 ID 109의 성격

ID 109는 데이터 품질 문제가 아니다. det 인스턴스 3,626개가 원본 defect 객체 3,413개에서 나와 비율이 1.06배이고, multipart 분리가 만든 조각은 213개로 5.9%에 불과하다. `min_extent_applied`는 0건이라 bbox 좌표 보정도 걸리지 않았다.

대형 공동을 제외한 뒤 기공 크기를 비교하면 109는 특별하지 않다. 중앙값 기공이 50,000 px 미만인 ID가 24개이고, 109의 annotation 이미지 1,437장 전부가 다른 ID들의 중앙 98% 범위(409~436,708 px) 안에 들어간다. 109의 특이점은 기공 크기가 아니라 이미지당 개수(2.52개 대 1.108개)다.

따라서 109를 제외하는 근거는 "다른 ID가 갖지 못한 유형이라 학습이 불가능하다"가 아니라 "fold 통계 안정성을 위해 annotation 22%를 포기한다"로 기록해야 정확하다.

### 26.7 채택한 해결책

이미지 단위 제외만으로는 일부 ID가 원본의 11~22%만 남는데, 이런 ID를 다른 ID와 같은 자격으로 fold에 넣으면 통계가 불안정하다. ID 단위 게이트를 §8.5로 신설한다.

- Gate 1: ID의 원본 이미지 중 `porosity_bbox_max_ratio >= 0.25`인 비율이 25% 이상이면 ID를 제외한다.
- Gate 2: Gate 1 생존 ID의 `annotations_per_image`에 Tukey far-out 울타리(Q3 + 3.0 x IQR)를 적용해 초과하는 ID를 제외한다.

두 게이트 모두 ID 이름을 직접 지정하지 않고 분포에서 임계값을 얻는다. 실제 데이터에서 Gate 1은 121, 122, 123, 124, 125, 126, 127, 131, 134를 제외하고 Gate 2는 109를 제외한다.

Gate 1의 임계값은 조건이 좋다. 오염률이 120의 22.7% 다음으로 135의 0.2%까지 떨어지므로 임계값을 1%와 22% 사이 어디에 두어도 같은 9개가 걸린다.

Gate 2는 한 번만 적용한다. 반복 적용하면 다음으로 밀도가 높은 ID가 계속 승격되어 데이터셋이 잠식된다. 실제 데이터에서는 109 제거 후 울타리가 1.6880이 되고 차순위인 135가 1.441이라 재적용해도 추가 제외가 없음을 확인했으나, 알고리즘 차원에서 반복을 금지한다.

Gate 2에는 함정이 하나 있다. defect를 가진 ID가 전체의 25% 미만이면 Q3와 Q1이 모두 0이 되어 울타리가 0으로 붕괴하고, annotation을 가진 모든 ID가 이상치로 잡힌다. IQR이 0이면 이상치가 정의되지 않으므로 게이트를 건너뛴다.

### 26.8 구조 일반화

ID 게이트가 몇 개를 제외할지 데이터에 따라 달라지므로 47/40/7 고정 구조를 버린다.

```text
ids_per_fold = (surviving_ids - 7) // 5
development  = ids_per_fold * 5
test         = surviving_ids - development
```

47개면 fold당 8개에 Test 7개로 기존과 같고, 37개면 fold당 6개에 Test 7개가 된다.

### 26.9 Test 선정 목적함수 보강

§9.4 목적함수에 annotations_per_image 격차를 추가한다. v3.7까지는 축, defect 비율, 면적 구간만 맞추고 밀도를 보지 않아 밀도 상위 ID가 Test에 몰렸다. 실측에서 Test 0.4963 대 development 0.2071로 2.4배 차이가 났는데, 이는 §11.3이 annotations_per_image를 도입해 막으려던 바로 그 불일치다.

### 26.10 검증 결과와 사전 추정의 괴리

2026-07-20 20:22 dry-run 실측이다. 37개 ID, Test 7 + development 30(fold당 6).

| 지표 | v3.7 실측 | 사전 추정 | v3.8 실측 |
|---|---|---|---|
| fold 최악 밀도 편차 | +52.46% (FAIL) | +8.26% | +19.20% (PASS, 여유 0.80%p) |

사전 추정은 Test 배정과 fold 배정을 하나의 스칼라 목적함수로 함께 최적화하는 별도 도구로 얻은 값이고, 실제 파이프라인은 greedy 배정과 stratum 제약 swap을 쓴다. 알고리즘이 다르므로 추정치를 구현 결과로 간주해서는 안 된다. 이후 설계 검토에서도 검증 도구의 탐색 방식이 구현체와 같은지 먼저 확인한다.

§9.4 목적함수에 추가한 annotations_per_image 격차는 의도대로 작동하지 않았다. 목적함수가 사전식 튜플이고 밀도 격차가 4순위인데, 앞선 축 격차와 defect 격차가 실수값이라 동점이 거의 발생하지 않아 밀도 항까지 비교가 내려오지 않는다. 그 결과 생존 ID 중 밀도 1위와 2위인 135와 102가 모두 Test에 배정되어 Test 밀도비가 2.40배에서 2.78배로 오히려 나빠졌다.

밀도 균형을 실제로 강제하려면 사전식 튜플 대신 가중 합 목적함수로 바꾸어야 하며, 이는 Test 선정 결과 전체를 바꾸므로 별도 검증이 필요하다. 현재 상태에서 하드 게이트는 통과하고 Test 밀도는 §17.2 게이트 대상이 아니므로, 데이터셋 생성 자체는 막히지 않는다.

### 26.12 2차 개선: Test 목적함수 스칼라화와 CT stratum 제약 해제 (2026-07-20 21:12)

26.10에서 드러난 두 문제를 실제 구현 위에서 다시 잡았다.

먼저 검증 방식을 바꿨다. 1차 개선의 사전 추정이 빗나간 원인은 별도 최적화기로 측정한 값을 구현 결과처럼 다뤘기 때문이다. 이번에는 manifest에서 CT 샘플 75,952장을 복원해 `_select_ct_test`, `_ct_fold_assignment`, `_swap_density` 를 그대로 호출하는 하네스를 만들고, 그 하네스가 직전 dry-run의 Test ID 집합과 fold 배정 30건을 완전히 재현하는지부터 확인했다. 재현이 확인된 뒤에만 가중치와 알고리즘을 건드렸다.

측정 결과 두 목표가 같은 원인으로 충돌한다는 사실이 드러났다. v3.8 초기 구현에서 fold 게이트가 통과했던 것은 설계 덕이 아니라, Test 선정이 밀도 상위 ID인 135와 102를 흡수해 development에서 치워준 부작용 덕이었다. Test 목적함수를 가중 합으로 바꿔 Test/development 밀도비를 0.947배로 되돌리자 그 ID들이 development에 남았고, 최악 fold 편차가 +26.93%로 게이트를 넘지 못했다.

fold 가중치로는 풀리지 않았다. 밀도 항 가중치를 4에서 40까지 올려도 +26.93%에서 내려가지 않았고, 40에서는 오히려 +61.95%로 악화됐다. 반면 별도 최적화기는 같은 37개 ID에서 +8.26%를 찾아냈다. 해가 존재하는데 배포된 탐색이 못 찾는 상태, 즉 목적함수가 아니라 탐색 품질의 문제였다.

원인은 §9.6의 stratum 동일 제약이었다. 밀도가 두꺼운 ID가 인구 희박한 stratum에 들어가면 교환할 상대가 없어 swap이 아무것도 하지 못한다. CT에서 이 제약을 해제하자 최악 편차가 +13.39%로 내려가 게이트를 6.61%p 여유로 통과했다. defect 균형은 swap 안의 ratio_spread 가드가 계속 지키므로 이탈이 0.0406에서 0.0426으로만 움직였다. RGB(§10.5)에는 제약을 그대로 둔다.

| 지표 | v3.7 | v3.8 1차 | v3.8 2차 |
|---|---|---|---|
| fold 최악 밀도 편차 | +52.46% (FAIL) | +19.20% (여유 0.80%p) | +13.39% (여유 6.61%p) |
| Test/development 밀도비 | 2.40배 | 2.78배 | 0.947배 |
| Test/development defect 비율 | - | 2.21배 | 1.001배 |
| fold defect 비율 최대 이탈 | 0.0148 | 0.0342 | 0.0426 |
| fold 이미지 비율 이탈 | 0.0333 | 0.0071 | 0.0089 |

작업 중 같은 실수를 두 번 했다. 검증 하네스가 `_swap_density` 를 직접 호출해 파이프라인 호출부의 설정을 타지 않았고, 그래서 이미 고친 코드를 두고 실패 수치를 읽었다. 설정이 호출부에 흩어져 있으면 검증이 다른 것을 잰다. CT fold 절차 전체를 `build_ct_folds()` 로 묶어 파이프라인과 하네스가 같은 함수를 쓰도록 바꿨다. 앞으로 CT fold 동작을 바꿀 때는 이 함수만 고친다.

### 26.11 남은 판단 사항

- Gate 2가 제외하는 ID 109는 미세 기공 annotation 3,626개를 갖는다. 미세 결함 검출이 목표라면 이 손실은 실질적이며, fold 통계 안정성과의 교환이라는 점을 인지하고 있어야 한다.
- Test 크기는 데이터에 따라 변한다. 생존 ID가 줄면 Test도 줄어 평가 신뢰도가 떨어지므로, 게이트 제외가 늘어나면 구조를 다시 검토해야 한다.
- 최적화 목적함수에 명시하지 않은 축은 반드시 희생된다. 이번 분석 중 Test 밀도를 목적에 넣지 않자 annotation이 거의 없는 ID가 Test로 몰렸고, defect 균형을 빼자 그쪽이 무너졌으며, 단일 ID 상한을 빼자 Test annotation의 99.7%를 ID 하나가 차지했다. 목적함수를 바꿀 때는 빠진 축이 무엇인지 먼저 확인한다.

## 27. 파이프라인 실행 런북과 자동화 제약 (2026-07-20 수정)

이 장은 파이프라인을 실제로 돌릴 때 반복해서 발목을 잡은 환경 문제와 그 해결책을 남긴다. 알고리즘이 아니라 실행 방식에 대한 기록이며, 버전이 올라가도 그대로 적용된다.

### 27.1 단일 진입점

`run_pipeline.ps1` 이 dry-run, approve, execute 세 단계의 유일한 실행 경로다. `run_dryrun_keepawake.ps1` 과 `run_execute_keepawake.ps1` 은 이 스크립트로 위임하는 얇은 래퍼로 남겨 두어 기존 호출 방식도 계속 동작한다.

```powershell
pwsh -File .\run_pipeline.ps1 -Stage dry-run -RawRoot "<raw>" -WorkDir "<work>" -Jobs 8 -Detached
pwsh -File .\run_pipeline.ps1 -Stage approve -WorkDir "<work>" -ApprovedBy "<검토자>"
pwsh -File .\run_pipeline.ps1 -Stage execute -RawRoot "<raw>" -WorkDir "<work>" -Output "<out>" -Jobs 8 -Detached
pwsh -File .\run_pipeline.ps1 -Stage upload  -WorkDir "<work>" -Output "<out>" -Remote "gdrive:<경로>" -Detached
```

`-Detached` 는 스크립트를 독립 프로세스로 다시 띄운다. dry-run과 execute처럼 30분을
넘기는 단계에는 반드시 붙인다. 아래 §27.4의 이유 때문이며, 인자는 스크립트가 배열로
넘기므로 경로의 공백도 안전하다.

실행 로직을 한 곳에 모은 이유는 v3.7 사고 때문이다. v3.6에서 v3.7로 올리면서 `run_dryrun_keepawake.ps1` 안의 모듈 이름이 `battery_v3_3.cli` 로 남아 있었고, 그 스크립트로 dry-run을 시도하면 `ModuleNotFoundError` 로 즉시 실패했다. execute 쪽 스크립트만 갱신되어 있었기 때문에 한동안 발견되지 않았다.

### 27.2 모듈 이름을 하드코딩하지 않는다

`run_pipeline.ps1` 은 스크립트 폴더에서 `cli.py` 를 가진 `battery_v3_*` 디렉터리를 찾아 모듈 이름을 결정한다. 버전을 올릴 때 패키지 폴더만 새 이름으로 바꾸면 실행 스크립트는 손대지 않아도 된다.

### 27.3 반드시 pwsh(PowerShell 7)로 실행한다

Windows PowerShell 5.1은 BOM 없는 UTF-8 파일을 ANSI 코드페이지로 읽는다. 이 저장소의 스크립트 주석과 데이터 경로에는 한글이 들어가므로, 5.1에서 실행하면 파싱 단계에서 문자열 종결자를 찾지 못하고 죽는다. 실제로 자체 래퍼를 5.1로 띄웠다가 다음 오류로 실패했다.

```text
문자열에 " 종결자가 없습니다.
    + FullyQualifiedErrorId : UnexpectedToken
```

두 가지로 대응한다. 저장소의 실행용 `.ps1` 은 UTF-8 BOM으로 저장하고, `run_pipeline.ps1` 첫머리에서 `$PSVersionTable.PSVersion.Major -lt 7` 이면 명확한 메시지와 함께 종료한다.

### 27.4 장시간 실행은 반드시 분리한다

에이전트 세션에서 실행한 자식 프로세스는 툴 호출이 끝나면 정리된다. dry-run은 약 45분, execute는 약 100분이 걸리므로 부모 세션에 묶인 채로 두면 산출물 없이 조용히 사라진다. 실제로 이 방식으로 두 번(약 78분) 헛돌았고, 종료 시 로그도 남지 않아 원인 파악에 시간이 더 들었다.

`run_pipeline.ps1 -Detached` 가 이 처리를 대신한다. 스크립트가 스스로를 독립
프로세스로 다시 띄우고 PID와 상태 파일 경로를 알려준 뒤 즉시 반환한다. 절차를 글로만
남기면 다음 사람이 다시 틀리므로 스위치로 만들었다.

### 27.5 경로의 공백

작업 폴더 상위 경로에 공백이 있으면 `-File` 인자가 그 지점에서 잘린다. 실제로 `-File 'C:\Users\rudtn\Desktop\Between'` 로 끊겨 실행이 실패했다. 인자는 위 예시처럼 배열로 전달하고, 문자열 하나로 이어 붙이지 않는다.

### 27.6 진행 상황 확인

파이프라인은 스캔이 끝난 뒤에야 보고서를 한꺼번에 쓰므로, 실행 중에는 work-dir이 비어 있는 것이 정상이다. 진행 여부는 다음으로 판단한다.

- `<WorkDir>\pipeline.status` 의 마지막 줄. `finished ... exit=0` 이면 정상 완료다.
- `Get-Process python` 의 워커 수와 CPU 시간. `--jobs 8` 이면 워커 8개가 각각 CPU를 쌓고 있어야 한다.

work-dir이 비어 있고 python 프로세스도 없으면 중단된 것이다.

### 27.7 병렬도

`--jobs` 는 scan과 output 단계의 워커 수다. Test/fold 선정과 보고서 작성은 단일 스레드이므로 `--jobs` 를 올려도 줄지 않는다. 6코어 12스레드 환경에서 8로 완주한 실적이 있으며, 12는 검증되지 않았다.

### 27.9 선정 로직 검증 도구

`tools/verify_ct_selection.py` 는 완료된 dry-run의 `manifest.csv`에서 CT 샘플을 복원해
파이프라인이 쓰는 선정 함수를 그대로 호출한다. 약 90초면 fold 편차, Test/development
밀도비, defect 균형을 볼 수 있어 37분짜리 dry-run을 반복하지 않아도 된다.

```powershell
python tools/verify_ct_selection.py "<work-dir>\reports" --check
```

`--check` 는 산출 결과가 그 work-dir에 기록된 배정과 일치하는지 확인한다. 선정 로직을
고치기 전에 이 검사를 먼저 통과시켜야 한다. 통과해야만 이 도구가 파이프라인을
대변한다고 말할 수 있고, 그 뒤의 측정값을 근거로 쓸 수 있다. §26.10과 §26.12의 두
사고가 모두 이 확인을 건너뛴 데서 나왔다.

### 27.8 산출물 업로드

execute는 마지막에 zip 4개를 CT trainval, CT test, EXT trainval, EXT test 순서로 하나씩 완성한다. 각 파일은 완전히 닫힌 뒤 다음이 시작되므로 완성된 것부터 업로드해도 된다.

완료 판정은 두 조건을 모두 만족할 때로 한다. 파일 크기가 일정 간격 두 번 동일하고, `zipfile.ZipFile` 로 열기가 성공해야 한다. 중앙 디렉터리가 기록되었는지만 보므로 19.7GB 파일에서 2.5초면 끝난다. `testzip()` 은 전체를 압축 해제하므로 쓰지 않는다.

zip은 무압축(`ZIP_STORED`)으로 만든다. 수록 대상이 PNG와 JPG라 deflate 이득이 CT 3.7%, EXT 1.2%에 그치는 반면, 처리량이 31~36 MB/s로 떨어져 93GB 기준 40분대의 단일 스레드 구간이 생긴다.
